## 1. Project Overview

### 1.1 Objectives
An automated WhatsApp customer support operations system that helps the Ops team manage support requests efficiently with AI assistance.

- Receive WhatsApp messages in real time
- Retrieve conversation context (transcript) and currently open tickets
- Call AI to classify requests (create ticket or ignore)
- Create tickets in Supabase, store conversation history, log analytics
- Send internal notifications (WhatsApp group) and email alerts
- Frontend Dashboard for Ops to monitor tickets, inbox, clients, and analytics

### 1.2 Tech Stack

| Component | Technology | Notes |
|---|---|---|
| Backend | Node.js / Express | REST API + WhatsApp ingestion |
| Frontend | Static HTML/CSS/JS | Premium dashboard, no build step required |
| Database | Supabase (PostgreSQL) | Cloud-hosted, realtime |
| AI Provider | Ollama (local) / Gemini | Configurable via .env, with fallback support |
| WhatsApp | whatsapp-web.js + puppeteer | Connected via QR code scan |
| Email | Nodemailer (Gmail) | Alert on new ticket creation |

## 4. Business Flow

### 4.1 Inbound Message Processing Pipeline

| # | Step | Details |
|---:|---|---|
| 1 | Receive WhatsApp message | Event from whatsapp-web.js: messageId, chatId, text, senderName |
| 2 | VIP Gate | VIP_MODE=strict: only process chatIds in vip_clients. allow_all: bypass filter |
| 3 | Duplicate Gate | Check if message_id was already processed via shouldContinueForMessageId() |
| 4 | Fetch Transcript | Retrieve the last N messages (N = WA_TRANSCRIPT_LIMIT) and build as text |
| 5 | Upsert Client | supabaseService.upsertClient(chatId, senderName) |
| 6 | Load Open Tickets | Retrieve top 5 open tickets for chatId to include in AI context |
| 7 | AI Decision | aiService.analyzeConversation() returns CREATE_SUBTASK or IGNORE |
| 8a | If CREATE_SUBTASK | Insert ticket, increment ticket_count, send WA alert + Gmail + auto-reply |
| 8b | If IGNORE | Log analytics event: conversation_ignored |
| 9 | Save Conversation | Insert into conversations: inbound + ai_decision + ticket_id |

### 4.2 Dashboard to Backend Flow
- Load data: Dashboard calls GET endpoints to render tickets, conversations, clients, and analytics
- Update ticket status: PUT /api/tickets/:id/status updates the status field
- Send message: POST /api/conversations/send triggers WhatsApp sendMessage + logs outbound conversation
- Add client: POST /api/clients inserts into vip_clients and upserts into clients

## 5. AI Engine

### 5.1 Providers

| Provider | Environment Variable | Endpoint |
|---|---|---|
| Ollama (local) | AI_PROVIDER=ollama | POST ${OLLAMA_BASE_URL}/api/generate |
| Google Gemini | AI_PROVIDER=gemini | Google Generative AI SDK |

### 5.2 Output Contract
The AI always returns a fixed JSON schema:
```json
{
  "decision": "CREATE_SUBTASK | IGNORE",
  "reason": "Reason for the decision",
  "summary": "Ticket title (if CREATE_SUBTASK)",
  "description": "Detailed description of the request",
  "priority": "High | Medium",
  "assignee_id": "Phuc_ID | Tram_ID | Vy_ID"
}
```

### 5.3 AI Context
- Chat transcript: capped at AI_TRANSCRIPT_LINES most recent lines (min 3, max 30, default 12)
- Open tickets context: top 5 open tickets (status not Done/Closed) for the chatId

### 5.4 Assignee Mapping

| AI Alias | Display Name | Role |
|---|---|---|
| Phuc_ID | Phuc (Engineering) | Handles technical issues |
| Tram_ID | Tram (Support) | Customer advisory and support |
| Vy_ID | Vy (Operations) | Internal and administrative matters |

### 5.5 Anti-Hallucination Measures
- Only real transcript data and real open tickets from Supabase are passed to the AI
- Force JSON output and normalize/validate the payload after receiving the response
- Strip code fences and extract the first JSON object if the model returns additional text
- Normalize decision to exactly two values: CREATE_SUBTASK or IGNORE

## 8. Operations Runbook

### 8.1 Initial Setup

#### Step 1: Set Up Supabase
- Create a new project at supabase.com
- Open the SQL Editor and run the full contents of `supabase/migration.sql`
- Copy your Project URL and Service Role Key into `.env`

#### Step 2: Configure .env
- Copy `.env.example` to `.env`
- Fill in `SUPABASE_URL` and `SUPABASE_SERVICE_ROLE_KEY`
- Choose `AI_PROVIDER` (ollama or gemini) and fill in the corresponding credentials
- Set `VIP_MODE` based on your requirements
- Optional: add Gmail credentials to receive email alerts

#### Step 3: Install Dependencies
```bash
npm install
```

### 8.2 Starting the System

#### Start Backend
```bash
npm start
```
- First run: A QR code will appear in the terminal. Scan it with WhatsApp on your phone to authenticate.
- The session is saved at `WA_AUTH_PATH`, so subsequent starts will not require re-scanning.

#### Start Frontend
```bash
node frontend/server.js
```
Access the Dashboard at:
```text
http://localhost:5000
```

### 8.3 System Verification
- Backend healthcheck: `GET http://localhost:3000/ping`
- Dashboard: `http://localhost:5000` — verify analytics data renders correctly
- Send a test WhatsApp message from a whitelisted number and confirm a ticket is created

## 9. Current Status & Known Issues

### 9.1 Idempotency (Duplicate Messages)
Issue:
- The `event_logs` table and `logMessageEvent` function exist but the pipeline does not currently write to `event_logs`.
- Idempotency via `event_logs` is therefore incomplete.

Recommended fix:
- Call `logMessageEvent(messageId)` immediately after a message passes the duplicate gate.
- `conversations.message_id` has a UNIQUE constraint, providing one layer of protection in the meantime.

### 9.2 Content-Based Ticket Deduplication
Current state:
- Uses a “soft rule”: open tickets context is included in the prompt so the AI avoids creating duplicates.
- No “hard rule” (hash or similarity check) exists at the technical level.

Risk:
- The AI may still create duplicate tickets if the same issue is phrased differently.

### 9.3 Anti-Hallucination
Already in place:
- Only real transcript data and real open tickets are used
- JSON output is enforced and the payload is normalized after receipt

Not yet implemented:
- Requiring AI to supply evidence quotes from the transcript
- Validating that the output content stays grounded in the actual transcript

### 9.4 Jira Integration
- JIRA_ASSIGNEE_*_ID variables remain in .env (legacy) but the backend does not call the Jira API.
- The jira_key field in the tickets table is reserved and never populated.

### 9.5 Frontend api.js
- `frontend/js/api.js` exists as an ES module but is not loaded by index.html.
- The Dashboard currently uses the inline apiService defined in app.js.
